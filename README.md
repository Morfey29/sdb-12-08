Задание 1. Резервное копирование
Кейс

Финансовая компания решила увеличить надёжность работы баз данных и их резервного копирования.

Необходимо описать, какие варианты резервного копирования подходят в случаях:

1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.

  Ответ: Для выполнения этого требования оптимально использовать комбинацию полного (full) и дифференциального (differential) резервного копирования:

  Полный бэкап выполняется, например, раз в неделю (воскресенье). Он содержит все данные на момент создания.

  Дифференциальный бэкап делается ежедневно и включает все изменения, произошедшие с момента последнего полного бэкапа. Таким образом, бэкап за среду содержит все данные, накопленные с воскресенья по среду включительно.

  Восстановление данных за предыдущий день (например, за вторник) потребует всего двух компонентов: последнего полного бэкапа и дифференциального бэкапа за вторник. Это обеспечивает приемлемую скорость восстановления и умеренный объём хранимых данных.

1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.

  Ответ: Здесь требуется более точный механизм, обеспечивающий низкий показатель RPO (Recovery Point Objective). Подойдёт стратегия полного бэкапа + непрерывное архивирование журналов транзакций (WAL или binary logs):

  Регулярно (например, раз в сутки) создаётся полный бэкап.

  База данных работает в режиме постоянной записи изменений в журналы транзакций. Эти журналы архивируются в надёжное хранилище с высокой периодичностью (каждые 5–10 минут или даже непрерывно).

  Восстановление до состояния «за час до поломки» выполняется так: разворачивается последний полный бэкап, а затем последовательно применяются все сохранённые журналы транзакций до нужного момента времени (Point-in-Time Recovery, PITR). Этот метод позволяет откатиться на любой момент с точностью до минуты/секунды.

1.3.* Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных.

Приведите ответ в свободной форме.

  Ответ: Да, такой сценарий реализуется с помощью репликации и механизмов отказоустойчивости (High Availability, HA):

  Настраивается репликация «ведущий-ведомый» (master-slave) или кластер с несколькими узлами. Все изменения с основной базы (мастера) в реальном времени передаются на одну или несколько реплик.

  Используется менеджер кластера, который непрерывно мониторит состояние мастера.

  При сбое мастера менеджер автоматически повышает одну из реплик до роли нового мастера (failover), а клиентские приложения перенаправляются на него (часто с помощью прокси-серверов типа HAProxy). Время переключения может составлять секунды, что для пользователя выглядит как кратковременная задержка.

  Важно: репликация не заменяет резервное копирование, так как логические ошибки (случайное удаление данных) мгновенно копируются на реплики. Однако она обеспечивает минимальное время простоя (RTO) и высокую доступность.

Задание 2. PostgreSQL

2.1. С помощью официальной документации приведите пример команды резервирования данных и восстановления БД (pgdump/pgrestore).

  Ответ:
  
  Резервное копирование (логический дамп):

```bash
bash
# Создание дампа базы данных "mydb" в пользовательском формате (сжатый, с возможностью выборочного восстановления)
pg_dump -h localhost -U postgres -F c -b -v -f /backup/mydb.backup mydb

    -F c — формат "custom" (рекомендуется для pg_restore).

    -b — включение больших объектов.

    -v — подробный вывод.
```
  Восстановление:

```bash
bash

# Предварительно создаём пустую базу (если её нет)
createdb -h localhost -U postgres newdb

# Восстанавливаем дамп в новую базу
pg_restore -h localhost -U postgres -d newdb -v /backup/mydb.backup

    -d newdb — целевая база данных.
```

Задание 3. MySQL

3.1. С помощью официальной документации приведите пример команды инкрементного резервного копирования базы данных MySQL.

  Ответ: Официальная утилита для горячего резервного копирования в MySQL - MySQL Enterprise Backup.
    Создание полного бэкапа:
       Ключевой информацией здесь является конечное значение LSN, которое записывается в метаданные этого бэкапа.
    
    ```bash
    bash

    mysqlbackup --user=root --password=*** \
       --backup-dir=/backups/mysql/full \
       backup-and-apply-log

        --backup-dir указывает директорию для сохранения файлов бэкапа.

        backup-and-apply-log создает полную резервную копию и подготавливает (применяет журналы) её для последующего восстановления.
    ```

  Создание инкрементного (или дифференциального) бэкапа:
    Для создания инкрементного бэкапа используется опция --incremental или --incremental-with-redo-log-only, а также опция --incremental-base, которая автоматически определяет LSN последнего бэкапа.

        Для создания дифференциального бэкапа (включает все изменения с момента последнего полного бэкапа):
        Параметр history:last_full_backup указывает утилите использовать самый последний полный бэкап в качестве базы для сравнения .
        
        ```bash
        bash

        mysqlbackup --user=root --password=*** \
           --backup-dir=/backups/mysql/diff_wednesday \
           --incremental \
           --incremental-base=history:last_full_backup \
           backup
        ```

        Для создания истинно инкрементного бэкапа (включает только изменения с момента последнего любого бэкапа — полного или инкрементного):
        Параметр history:last_backup указывает утилите найти LSN последнего успешного бэкапа (неважно, полного или инкрементного) .
        
        ```bash
        bash

        # Первый инкрементный бэкап после полного
        mysqlbackup --user=root --password=*** \
           --backup-dir=/backups/mysql/inc1 \
           --incremental \
           --incremental-base=history:last_backup \
           backup

        # Второй инкрементный бэкап (будет содержать изменения только с момента inc1)
        mysqlbackup --user=root --password=*** \
           --backup-dir=/backups/mysql/inc2 \
           --incremental \
           --incremental-base=history:last_backup \
           backup
        ```

  Подготовка к восстановлению:
    Чтобы восстановить данные из цепочки "полный бэкап + инкрементные", их необходимо объединить. Для этого последовательно применяются инкрементные бэкапы к полному.

    ```bash
    bash

    # Применяем первый инкрементный бэкап к полному
    mysqlbackup --backup-dir=/backups/mysql/full \
       --incremental-backup-dir=/backups/mysql/inc1 \
       apply-incremental-backup

    # Затем применяем второй инкрементный бэкап
    mysqlbackup --backup-dir=/backups/mysql/full \
       --incremental-backup-dir=/backups/mysql/inc2 \
       apply-incremental-backup
    ```

    После выполнения этих команд директория /backups/mysql/full будет содержать полные, актуальные данные на момент создания последнего инкрементного бэкапа и готова к восстановлению .

3.1.* В каких случаях использование реплики будет давать преимущество по сравнению с обычным резервным копированием?

Приведите ответ в свободной форме.

   Использование реплики (ведомого сервера) дает ряд стратегических преимуществ, которые напрямую не обеспечиваются простым резервным копированием. Официальная документация MySQL выделяет следующие ключевые сценарии :

   Выполнение резервного копирования без влияния на основной сервер (Offloading backups): Это, пожалуй, самое важное преимущество. Создание резервной копии — ресурсоемкая операция, которая может создать значительную нагрузку на диск и процессор. Выполняя бэкап на реплике, вы полностью исключаете негативное влияние на производительность основного сервера (источника), который обслуживает критически важные запросы приложений . Это позволяет создавать бэкапы в любое время, не беспокоясь о простое.

   Обеспечение высокой доступности и аварийного переключения (High Availability): Реплика является "горячей" копией данных. В случае сбоя основного сервера (мастера) реплика может быть быстро повышена до его статуса, и приложения смогут продолжить работу с минимальным временем простоя (RTO). Восстановление из традиционного бэкапа заняло бы значительно больше времени .

   Распределение нагрузки на чтение (Scaling out reads): Реплики можно использовать для обработки запросов на чтение данных (SELECT), особенно в системах, где количество операций чтения значительно превышает количество операций записи. Это снижает нагрузку на основной сервер и повышает общую производительность и отзывчивость системы .

   Аналитика и отчетность (Data Analysis): Можно направлять сложные и тяжелые аналитические запросы или задачи по созданию отчетов на реплику. Это гарантирует, что подобные операции не замедлят работу основного сервера, обрабатывающего оперативные транзакции .

   Важно подчеркнуть, что репликация не является заменой резервному копированию . Реплика защищает от сбоя оборудования, но не от логических ошибок. Если на мастере будет случайно выполнена команда DROP DATABASE, она мгновенно реплицируется на все ведомые серверы, уничтожая данные везде. Для восстановления на такой момент времени (Point-in-Time Recovery) потребуется именно резервная копия, созданная до ошибки.
